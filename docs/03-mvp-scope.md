# 03 — MVP Scope (Phase 1)

The **Telegram Approval Bridge**. The smallest thing that delivers the core value.

## What v1 does

1. Listens for Claude Code's `PreToolUse` HTTP hook.
2. Classifies the requested command by risk using a configurable pattern list.
3. If risky, sends a Telegram message to the configured user with **[Approve] [Reject] [Modify]** buttons.
4. Waits up to 9 minutes (safely under Claude Code's default 600s hook timeout).
5. Returns the user's decision to Claude Code so it proceeds, aborts, or runs a modified command.

That's it.

## What v1 does NOT do

- ❌ No voice notes (Phase 2)
- ❌ No screenshots (Phase 2)
- ❌ No localhost tunneling (Phase 3)
- ❌ No multi-agent `@` routing (Phase 6)
- ❌ No web dashboard (Phase 4)
- ❌ No multi-machine hub (Phase 5)
- ❌ No tmux UI through Telegram (later phase, exact placement TBD)
- ❌ No support for non-Claude-Code agents (Codex/Aider come in Phase 6)
- ❌ No voice/AR (Phase 7)

If you find yourself adding any of these, you are out of scope. Stop. Add it to a later phase.

## Acceptance criteria

A user with a fresh checkout can:

1. Run `pnpm install` then `pnpm build`.
2. Run `npx wazir init` and answer prompts (bot token, allowed Telegram chat ID, hub port, worker port).
3. Run `wazir hub` in one terminal.
4. Run `wazir worker` in another terminal.
5. Paste the snippet that `wazir init` prints into `~/.claude/settings.json` (configures the `PreToolUse` HTTP hook).
6. Start a Claude Code session in any repo; ask it to run `git push`.
7. Within 3 seconds, see a Telegram message:
   > **Approve git push?**  
   > `git push origin main`  
   > `@ macbook-fay : ~/Projects/...`  
   > **[✅ Approve] [❌ Reject] [✏️ Modify]**
8. Tap **Approve** → Claude Code proceeds, push happens.
9. On the next risky command, tap **Reject** → Claude Code reports the action was denied.
10. On the next risky command, tap **Modify**, reply with the modified command, send → Claude Code runs the modified command.
11. Run `wazir status` → see hub/worker health and the last 5 approvals with their decisions.

That's the deliverable. Everything else is scope creep.

## Default risk patterns

The worker classifies these as risky by default (user can edit in config):

```yaml
risk_patterns:
  - name: rm_force
    regex: '\brm\s+(-[a-z]*r[a-z]*f|-[a-z]*f[a-z]*r)\b'
    label: "rm -rf"
  - name: rm_plain
    regex: '\brm\s+'
    label: "rm"
  - name: git_push
    regex: '\bgit\s+push\b'
    label: "git push"
  - name: git_push_force
    regex: '\bgit\s+push\s+.*--force\b|\bgit\s+push\s+.*-f\b'
    label: "git push --force"
  - name: git_reset_hard
    regex: '\bgit\s+reset\s+--hard\b'
    label: "git reset --hard"
  - name: git_clean
    regex: '\bgit\s+clean\b'
    label: "git clean"
  - name: sudo
    regex: '\bsudo\b'
    label: "sudo"
  - name: publish
    regex: '\b(npm|yarn|pnpm)\s+publish\b'
    label: "package publish"
  - name: migration
    regex: '\b(prisma\s+migrate|knex\s+migrate|sequelize\s+db:migrate)\b'
    label: "migration"
```

If none match, the command runs without approval.

## Non-functional requirements

| Requirement | Target |
|---|---|
| Telegram round-trip latency | < 3s typical |
| Hook timeout headroom | 9 min (Claude Code allows 10) |
| Process restart resilience | Hub and worker restart cleanly; in-flight approvals time out gracefully |
| Disk footprint | < 50 MB installed (excluding `node_modules`) |
| Memory footprint | < 100 MB per daemon idle |
| Logs | structured JSON, rotated daily |
| Setup time for new user | < 5 minutes from `pnpm install` to first approved command |

## Out-of-band items handled by Phase 1

These are minor but explicitly in scope because they make the MVP usable:

- `wazir init` wizard generates the Telegram allowlist by sending a "say hi" message from the bot and capturing the chat ID of whoever replies.
- The Claude Code hook config snippet is generated by `wazir init` and written to a printable file (not auto-injected) — the user copies it to `~/.claude/settings.json` themselves. Mutating user config is out of scope for v1.
- Approval history is browsable via `wazir log` (CLI command). No web UI.
